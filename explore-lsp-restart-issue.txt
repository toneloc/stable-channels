LSP RESTART CRASH-LOOP — LSPS2 PERSISTED STATE FAILS TO RELOAD
==============================================================
Date discovered: 2026-07-18 (e2e regtest stack, branch autotest)
Status: e2e FIXED (surgical state delete); MAINNET LSP STILL EXPOSED
Severity: HIGH for prod — next restart of the live LSP may not come back up.

SYMPTOM
-------
ldk-server crash-loops at startup with exactly one uninformative line:

    ERROR ldk_server:259] Failed to build LDK Node: Failed to read from store.

immediately after ALL channel state loads successfully (the log shows every
ChannelMonitor "Applying update ..." and ChannelManager "Successfully loaded at
update_id N against monitor at update id N" line, then the error ~7ms later).
Nothing else is logged. bitcoind/electrs reachable, disk fine, sqlite readable.

Trigger in e2e: the 2026-07-18 container rename recreated the ldk-server
container — its FIRST restart since 2026-07-16 01:44 (it had run continuously
through 2.5 days of JIT-onboard + splice-in/out flows). The restart itself was
clean ("Shutting down LDK Node ..." logged at 21:04:59); the store was not
corrupted by the shutdown. The state it had accumulated while running is what
would not load back.

ROOT CAUSE
----------
Versions (identical e2e and prod):
  ldk-server @ 0e4434d7083ae9926c73f26e7cd52f00bde44d37
  ldk-node   @ f2e44fdd21490ff3fa90445d7a28b19cfd020c3d  (pinned by ldk-server)
  built with --features experimental-lsps2-support, acting as LSPS2 service.

Two compounding problems:

1) SILENT ERROR PATH (why it was so hard to diagnose).
   ldk-node src/liquidity/mod.rs:283 (rev f2e44fd):

       LiquidityManager::new(... kv_store ...)
           .await
           .map_err(|_| BuildError::ReadFailed)?,

   The real error is DISCARDED — no log_error, no detail. Every other
   BuildError::ReadFailed site in builder.rs logs a specific message first
   ("Failed to read channel monitors / network graph / scorer / event queue /
   peer data / payment data ..."); this is the only silent one. So a bare
   "Failed to read from store." with all channel state loading fine points
   HERE: the LiquidityManager reload.

2) THE ACTUAL FAILURE: persisted per-peer LSPS2 service state doesn't
   re-deserialize. LiquidityManager::new reads its persisted service state
   from the node KV store:

       ldk_node_data.sqlite, table ldk_node_data
       primary_namespace = 'lightning_liquidity_state'
       secondary_namespace = 'lsps2_service'
       key = <counterparty node id>  (one row per peer, 246-792 bytes each)
       (plus one 'event_queue' row in the same primary namespace)

   The e2e store had ~34 such rows accumulated from JIT-open and splice flows.
   At least one of them errors on read-back, failing the whole build.

   Strong hint at how the bad state got written — the runtime log repeatedly
   shows the LSPS2 state machine being driven into an unexpected state by
   splice-triggered ChannelReady events:

       ERROR ldk_node::liquidity::service::lsps2:200] LSPS2 service failed to
       handle ChannelReady event: Misuse error: Failed to transition to channel
       ready: Channel ready received when JIT Channel was in state:
       PaymentForwarded { channel_id: fb1ba2... }

   i.e. a splice on a JIT channel re-fires ChannelReady, the LSPS2 handler
   rejects it (JIT channel already in PaymentForwarded), and whatever state was
   persisted around these sequences is what later fails to reload. (Exact
   failing field not isolated — the swallowed error makes that impossible from
   logs alone; would need the state blobs run through the deserializer under a
   debugger, or the map_err replaced with a logging version.)

EVIDENCE TRAIL (how this was pinned down)
-----------------------------------------
- /data/regtest/ldk-server.log*: single boot 2026-07-16 01:44 -> clean shutdown
  2026-07-18 21:04:59, then crash-loop from 21:05:11 onward. Docker restart
  backoff visible (0.5s -> 60s intervals).
- All ReadFailed sites in builder.rs (rev f2e44fd) log before returning —
  except the liquidity one. No pre-error log line => LiquidityManager::new.
- sqlite inspection showed the lightning_liquidity_state/lsps2_service rows;
  deleting them fixed the boot instantly, everything else untouched.

FIX APPLIED (e2e, 2026-07-18 ~21:43Z) — VERIFIED WORKING
--------------------------------------------------------
    cd e2e/harness
    docker compose stop ldk-server
    # inside the volume (alpine + sqlite3):
    cp /data/regtest/ldk_node_data.sqlite /data/regtest/ldk_node_data.sqlite.bak-lsps2
    sqlite3 /data/regtest/ldk_node_data.sqlite \
      "DELETE FROM ldk_node_data WHERE primary_namespace='lightning_liquidity_state';"
    docker compose start ldk-server

Result: node boots with the SAME node id (020ebf6f...f7b7), all channels and
funds intact, peers reconnect, sc-lsp gRPC reconnects. Backup left in the
volume as ldk_node_data.sqlite.bak-lsps2.

What is lost by deleting: the LSPS2 service's per-peer JIT bookkeeping —
in-flight JIT open promises (fee params quoted, intercept SCIDs, pending
channel-open state). Channels that already exist are NOT affected (they live in
ChannelMonitor/ChannelManager state). Risk window: a client mid-JIT-onboard at
the moment of deletion would have its pending open forgotten and would need to
retry. When no JIT open is in flight, deletion is effectively free.

PROD EXPOSURE — WHY THIS MATTERS BEYOND E2E
-------------------------------------------
The live mainnet LSP (bidaemon pair on the LSP server: stable-channels-lsp +
backend ldk-server @ 0e4434d / ldk-node f2e44fdd) is the SAME code with the
SAME LSPS2 service config (e2e config is prod-parity by design). It has real
accumulated JIT state from real user onboards, and it has NEVER restarted
(NRestarts=0 — established during the July force-close investigation). Splices
happen on prod JIT channels too (splice-in on deposit is a core flow).

=> The next prod restart — planned upgrade, crash, host reboot, OOM — may hit
   this exact crash-loop, with the LSP DOWN and nothing in the logs but
   "Failed to read from store." That is an availability incident for every
   wallet user (no JIT opens, no stability engine, no proxy) until someone
   applies the remedy.

RUNBOOK — if/when the prod LSP won't come back up with this signature
---------------------------------------------------------------------
Signature check: all "Successfully loaded ... against monitor" lines present,
then "Failed to build LDK Node: Failed to read from store.", nothing between.

1. Stop ldk-server (leave stable-channels-lsp down too; it needs the backend).
2. Back up the node sqlite in the ldk-server data dir (network subdir, e.g.
   bitcoin/ldk_node_data.sqlite):  cp ldk_node_data.sqlite ldk_node_data.sqlite.bak-$(date +%F)
3. sqlite3 ldk_node_data.sqlite \
     "DELETE FROM ldk_node_data WHERE primary_namespace='lightning_liquidity_state';"
4. Start ldk-server; confirm "Starting up LDK Node with node ID <expected id>".
5. Start stable-channels-lsp; confirm gRPC connect + REST up + price medians.
6. Expect: any client that was mid-JIT-onboard retries its request.

MITIGATION WITHOUT UPSTREAM — SELF-HEALING GUARD (added 2026-07-19)
-------------------------------------------------------------------
umbrel/docker/ldk-server-guard.sh wraps the ldk-server process: on 3
consecutive fast exits bearing the bare "Failed to read from store."
signature (only the LSPS2 reload fails without a companion detail line), it
backs up ldk_node_data.sqlite and deletes the lightning_liquidity_state
namespace, then lets the next start proceed. Healthy restarts lose nothing.
Deployed in both Docker images (e2e + Umbrel package).

Prod (systemd on the LSP box) — install during the controlled restart:
  1. scp umbrel/docker/ldk-server-guard.sh to /home/ubuntu/bidaemon/
  2. In the ldk-server unit, replace ExecStart with:
       ExecStart=/home/ubuntu/bidaemon/ldk-server-guard.sh \
         /home/ubuntu/bidaemon/ldk-server-config.toml \
         /home/ubuntu/bidaemon/bidaemon-ldk-data
     (network dir auto-derived from the config; ensure sqlite3 is installed)
  3. The guard loops internally — set the unit's Restart=no so systemd's
     restart counter doesn't fight the wrapper's.
  4. daemon-reload + restart per the runbook below, runbook staged.

PROACTIVE OPTIONS FOR PROD (pick one)
-------------------------------------
a) Scheduled controlled restart during a quiet window WITH the runbook staged,
   to find out now rather than during an incident. If it crash-loops, apply
   steps 1-6 (minutes of downtime, at a time of our choosing).
b) Pre-clean: stop, backup, delete the lightning_liquidity_state rows, start.
   Guarantees the reload bug can't bite, at the cost of forgetting any
   in-flight JIT opens (verify none pending first via audit log / recent
   JIT activity).
c) Wait for upstream fix and upgrade ldk-server/ldk-node past it, then restart.
   Leaves the landmine armed until then.

UPSTREAM FILING (ldk-node / lightning-liquidity)
------------------------------------------------
Two issues worth filing:
1. ldk-node: BuildError::ReadFailed from LiquidityManager::new swallows the
   underlying error (src/liquidity/mod.rs:283 @ f2e44fd) — should log_error the
   source like every other ReadFailed site. Trivial fix, huge diagnosability win.
2. lightning-liquidity / ldk-node LSPS2 service: persisted per-peer lsps2_service
   state written during splice-on-JIT-channel sequences fails to deserialize on
   restart, bricking node startup. Repro context: LSPS2 service with
   client_trusts_lsp=true, JIT open -> payment forwarded -> splice_init on the
   same channel (the "Channel ready received when JIT Channel was in state:
   PaymentForwarded" misuse errors mark the write-side of the corruption).
   Attach: the backed-up sqlite's lightning_liquidity_state rows from the e2e
   volume (regtest, no privacy concerns) — ldk_node_data.sqlite.bak-lsps2 in
   the harness_ldk-server-data docker volume.
   Note both are experimental features (experimental-lsps2-support, splicing),
   but the failure mode (node won't start) is worse than a degraded service.
